# v3 ingestion prompt: model intermittently drops requested fields

**Date observed:** 2026-05-02
**Status:** mitigated (validator now strict-requires only the load-bearing fields; `simpler_version` is auto-repaired via a focused follow-up LLM call; other dropped fields degrade gracefully)
**Severity:** medium
**Area:** generator (LLM prompt / model behaviour)

## Symptom

The v3 ingestion prompt requests a structured object with several fields (`normalised_form`, `topic_tags`, `difficulty`, `opening_prompt`, `full_solution`, `milestone_answers`, `simpler_version`, `explain_it_back`). Across generations, the model intermittently drops different fields. Observed in validation:

- `milestone_answers` missing — request failed with "v3 response missing field: milestone_answers"
- `simpler_version` missing — request failed with "v3 response missing field: simpler_version"

Each happened on a different fresh problem submission. Both fields are mentioned in the prompt's field list and shown in the worked examples; the model still drops them.

## Reproduction

Submit fresh problems at `/homework` with v3 of the `ingestion` prompt active. Failure is intermittent and varies by problem.

## Mitigations in place (2026-05-02)

**1. `simpler_version` follow-up call.** When the main v3 generation produces a response without a usable `simpler_version`, the generator fires a small focused LLM call (`SIMPLER_VERSION_SYSTEM_PROMPT` in `gcse_help_prompts.py`) that takes the original problem and `full_solution` and returns just the simpler-version object. The result is merged into the main response *before caching*, so cached responses always carry a simpler version even when the main generation dropped it. Failure of the follow-up call is non-fatal — `simpler_version` simply stays absent and the *Simpler version* button remains disabled for that problem.

Cost: one extra ~600-token call per affected problem, once. After caching, free.

**2. Loose validator.** `_validate_v3_response` now strict-requires only:

- `normalised_form` — needed to display the question
- `full_solution` — needed for the evaluator's LLM-path grounding

Everything else (`opening_prompt`, `milestone_answers`, `simpler_version`, `explain_it_back`, `topic_tags`, `difficulty`) is treated as optional. If present and malformed (wrong type, empty entries) it still raises; if absent the validator logs a WARNING and proceeds.

The frontend already has graceful fallbacks for each:
- Missing `opening_prompt` → generic fallback line
- Missing `milestone_answers` → cheap-path can't fire (every submission goes through the LLM evaluator path); functionally working, pedagogically degraded
- Missing `simpler_version` → "Simpler version" mode toggle button stays disabled with the "not available" tooltip
- Missing `explain_it_back` → not rendered anywhere in the current UI; safe to omit
- Missing `topic_tags`/`difficulty` → not visually prominent

## Possible proper fixes

In rough order of cost:

1. **Strengthen the prompt with stricter language and ordering.** Move the most-frequently-dropped fields earlier in the spec. Use explicit "**You MUST include `<field>`. If you cannot, use a sensible default rather than omitting.**" Cheap, may not fully solve.
2. **Post-process to derive missing fields from `full_solution` / `normalised_form`.** Specifically:
   - `milestone_answers`: split `full_solution` on sentence boundaries and extract `= X` patterns. Take the final one as the canonical answer.
   - `opening_prompt`: derive from `topic_tags[0]` if available, e.g. "It's a {tag} problem — where would you start?".
   - `simpler_version`: hard to derive without a second LLM call; this one stays optional.
3. **Two-pass generation for the dropped fields.** Cheap follow-up call to fill in whichever fields are missing. Adds latency.
4. **Stronger model.** `gpt-4o` follows structured output more reliably than `gpt-4.1-mini`. Cost implication.

(2) for `milestone_answers` and `opening_prompt` is the most reliable cheap fix. `simpler_version` is harder; if it's dropping consistently, escalate to (4).

## Workaround

Failing problems can be regenerated by toggling `useCache=false` (or waiting for cache to age). Most failures are non-deterministic; a regeneration usually produces a complete response.
